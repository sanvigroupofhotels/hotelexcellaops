# HEOS Core v1.0 — Business Workflows

Operational lifecycles. Each workflow lists trigger, states, engines,
and side effects.

## Booking Lifecycle

```
lead → enquired → quoted (deprecated) → confirmed →
       arrived (check-in) → in_house →
       checked_out
              ↘ cancelled | no_show
```

- **Confirmed.** Written by `booking-create.ts`. Deposit (if any) via
  `booking-payments-api.ts`. Emits `booking.confirmed`.
- **Check-in.** `check-in-flow.tsx` orchestrator: verifies documents,
  assigns rooms, collects balance, updates `bookings.status`. Emits
  `booking.checked_in`.
- **In-house.** In-house charges via `booking-charges-api.ts`. Extension
  and room-change flows update `booking_room_assignments`.
- **Checkout.** `booking-stay.ts` finalizes, then `hk-checkout-hook.ts`
  enqueues HK tasks per assigned room. Emits `booking.checked_out`.
- **Cancel.** Refund via payments engine; state transitions logged.

## Guest Journey

```
Booking Engine → confirmation → email/WA with portal token →
Portal (pay balance, upload docs, request extension/room-change) →
Check-in → In-house → Checkout thank-you → Review request
```

## Housekeeping Lifecycle

```
task_created (checkout hook | manual | service) →
claimed (housekeeper) → completed →
[optional: linen returned to laundry_queue] →
room ready
```

- Auto-generated by `hk-generator.ts` on checkout.
- Manual/service tasks via `hk-tasks.ts`.
- Completion writes linen counts to `laundry_queue` when relevant.
- Emits `hk.task_completed`, `hk.issue_reported`.

## Laundry Lifecycle

```
laundry_queue (populated by HK) →
batch_created (vendor selected) →
sent → returned → reconciled
              ↘ damaged | lost
```

- `laundry-batches-api.ts` moves queue rows into batch lines.
- Vendor selected from `vendors` master.
- Loss/damage writes to `laundry_batch_lines.status` and emits event.

## Inventory Lifecycle

```
item_created (master) →
movement_in (purchase from vendor) →
movement_out (consumption / adjustment) →
stock recomputed
```

- All movements append to `inventory_movements`; running stock derived.

## Cash Book Lifecycle

```
Business-Date start (00:00 IST for ops purposes) →
transactions (in/out) →
end-of-day close (Night Audit) →
snapshot in cash_audit_closes
```

- Transactions are Business-Date-scoped, not calendar-date.
- Closing a day is idempotent; re-close creates a new audit row.

## Night Audit Lifecycle

```
day begins → operational activity → critical tasks reviewed →
EOD report generated → decisions recorded → business_date advanced →
next day begins
```

- `perform-night-audit.ts` orchestrates the entire sequence.
- `app_settings.business_date` is bumped exactly once.
- Trigger blocks advancing past calendar date.

## Business Date Lifecycle

```
09:00 (or configured) — new business_date begins
→ all operational tables filter by business_date
→ Night Audit advances at end of ops day
→ never exceeds calendar date (Asia/Kolkata)
```

Enforcement:

- `app_settings_guard_business_date` trigger.
- Every operational query uses `current_business_date()` or reads
  `app_settings` directly through engine helpers.
- Reporting date ranges default to Business Date, with calendar-date
  fallback for backward-looking reports.

## Complaint Lifecycle

```
opened (guest/ops) → assigned → in_progress →
resolved | escalated
```

## Booking Payment / Razorpay Lifecycle

```
order_created (Ops or Portal) → hosted checkout →
webhook received (signature verified) →
payment recorded in booking_payments →
booking balance recomputed →
notifications dispatched
```

- `/api/public/razorpay-webhook` is the only external write path;
  every event stored in `razorpay_webhook_events` before mutation.

## Extension / Room Change

```
request (Portal or Ops) → availability check (room-availability.ts) →
booking_room_assignments updated →
booking dates recomputed →
pricing recomputed →
balance updated →
guest re-notified
```
